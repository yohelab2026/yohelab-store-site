# yohelab-store-site 運用ルール

このリポジトリをCodexでメンテするときの基準。

## 最優先
- 目標は `文標` を主力商品として育てること。
- 既存の売れる導線を壊さない。
- 誇大表現は使わない。
- `AI検索に出る` とは書かない。
- `AI検索時代の記事構造` は使ってよい。

## 変更方針
- 変更は小さく、意図が分かる単位で行う。
- 本番文言、価格、Stripe URL、法務文言は勝手に変えない。
- 旧商品名や旧導線は、残すなら互換目的だけにする。
- 主要な公開文言は `よへラボ` と `文標` を優先する。

## LINE / AI自動化の線引き
- 自動で進めてよい: 文章、デザイン、LP、ブログ、軽いUI修正、リンク切れ、画像最適化、誤字、アクセシビリティ、速度改善。
- 必ず人間確認: 価格、Stripeリンク、決済、法務、返金、プライバシー、利用規約、特商法、文標ZIP本体、子テーマZIP、シリアル、ライセンス認証、Cloudflare Functionsの認証/決済処理、GitHub Actions、Secrets、デプロイ設定。
- `AI作業OK` のIssueでも、確認必須領域を触るPRは自動マージしない。
- 確認必須領域を触ったPRには `needs-human-review` を付け、理由をコメントする。
- 安全領域だけのPRは `safe-auto-candidate` として扱い、テスト成功後に自動マージ候補にできる。
- 「戻して」と言われたら、直前のAI変更はrevert PRで戻す。履歴を壊すforce pushやresetは使わない。

## ブログ画像の自動作成ルール
- LINEでブログ作成・リライト時に画像指定がある場合は、記事本文と一緒に画像案も作る。
- 画像は著作権のある素材を拾わず、オリジナルの図解、SVG、WebP、HTML生成の軽いビジュアルを優先する。
- 画像ごとに `alt`、挿入位置、画像の目的を明記する。
- 本文に埋め込む画像は、できるだけ `width` / `height` / `loading="lazy"` / `decoding="async"` を付ける。
- 画像が作れない環境では、代わりに軽いSVG図解かブログカード風の静的ビジュアルを作り、PR本文に理由を書く。
- 「実機テスト」「開発確認」などの売り手・内部向け文言を画像内に出さない。読む人・買う人に向けた言葉だけにする。

## 毎回確認すること
- トップ、LP、デモ、更新履歴、問い合わせの文言が矛盾していないか。
- `.htaccess`、シリアル、更新、法務の表記が一致しているか。
- スマホ表示で崩れないか。
- 画像、動画、ロゴのリンク切れがないか。
- FAQ、構造化データ、CTA が壊れていないか。

## 守ること
- Lighthouse、速度、アクセシビリティを落とさない。
- `smoke-test.mjs` が通る状態を維持する。
- 画像は重くしすぎない。
- ブログの導入メモや更新履歴は、購入前の安心材料として扱う。

## 作業ルール
- 手動編集は `apply_patch` で行う。
- 変更後は `npm run build` と `npm run test:smoke` を通す。
- 触っていないファイルは戻さない。
- 迷うときは、商品説明よりも購入者の安全を優先する。
